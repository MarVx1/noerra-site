# ADR 0001: Полностью rule-based Editorial Engine, без LLM

## Статус

Принято, действующее.

## Контекст

Генерация текста статей (перевод абстракта → структура → черновик поста)
могла бы использовать LLM (Anthropic/OpenAI и т.п.) для смысловой работы:
идея-экстракция, стиль, аналогии, subjective quality check. Ранний вариант
Editorial Engine это рассматривал.

## Решение

Весь текстовый пайплайн — regex, шаблонные банки (`analogy_bank.py`,
`transitions.py`, `reader_question.py`), эвристики классификации сценария
и декомпозиции абстракта. Никаких вызовов LLM API нигде в проекте.

## Причина

Финансовая: у пользователя нет бюджета на LLM API costs. Это не техническое
предпочтение, а жёсткое ограничение — подтверждено явно и повторно.

## Последствия

- Есть признанный, осознанно принятый потолок качества: тяжёлые
  академические предложения на 30-50 слов не упрощаются семантически,
  блок «Ограничения» шаблонный, нет настоящей Idea Extraction, «момент
  понимания» не создаётся, Human Check субъективных вещей невозможен.
- Это не баги и не открытые задачи — их не чинить в рамках текущего
  бюджета.
- Компенсирующий механизм — ручная вычитка на реальных данных
  (`adaptation/content_audit.py` автоматизирует только уже встречавшийся,
  узкий класс структурных дефектов, не смысловые).
- Любое предложение добавить LLM-интеграцию в этот проект должно
  исходить от пользователя явно, а не быть инициативой со стороны.
